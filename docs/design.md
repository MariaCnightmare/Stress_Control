# Stress_Control - Design Notes

## 基本思想
Stress_Control は「最適化ツール」ではなく **判断支援ツール**である。
OS を直接制御する自動最適化は、文脈（ユーザーの意図・作業状態）を失う危険があるため採用しない。

## 目的
- システムの負荷状態を観測し、整理された判断材料として提示する
- 高負荷の「継続」や「悪化傾向」を見える化し、対処の意思決定を助ける
- Windows を主対象としつつ、WSL(Ubuntu) でも同一コードで動作させる

## 禁止事項（安全制約）
- 自動 kill / 自動タスク停止
- 自動 renice / 自動優先度変更
- 管理者権限を前提とする自動変更（サービス改変、レジストリ改変等）
- 初期段階での常駐化（段階導入する場合は設計レビューを必須とする）

## ストレスの定義（本プロジェクト内）
「ストレス」とは、以下のような状態を指す：
- CPU/メモリ等の高負荷が継続する
- スパイク（瞬間）ではなく持続によって作業の阻害が起きる
- 作業状態とシステム状態が乖離している（例：体感が重いのに数値は低い/その逆）

## 実装方針
- コアは分析ロジック（観測・集計・判定）
- UI（Web/TUI）は後付け
- 出力JSONの互換性を維持し、report_versionで進化させる

## Process Exclusions
- Windowsの "System Idle Process"(pid 0) はCPU空き時間を示すため解析対象から除外する。
- 解析の自己影響を避けるため、実行中の stress-control 自身のプロセスはデフォルトで除外する。
